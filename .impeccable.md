# Cloudstash Design Context

## Users

Cloudstash is a personal link-saving tool with AI summaries, authored for a technically fluent power user who ingests many links per day from multiple sources (web paste, Telegram bot, Raycast, iOS Shortcut, future Chrome extension) and returns to them later to read, chat with, or recall. The design should assume keyboard fluency, multi-device use, and a low tolerance for sluggishness. Polish should be sufficient that the tool could grow into a small product for like-minded users without a redesign.

## Brand Personality

**Precise. Light. Premium.**

- **Precise** — every element is deliberate; no padding "just because", no decoration that doesn't carry meaning. Reads as engineered, not assembled.
- **Light** — literally (light-mode first) and figuratively (weightless, breezy, never gloomy). No neon, no dark cave, no heavy shadows.
- **Premium** — not ornate. Premium through restraint and execution: confident whitespace, careful weights, details that reward attention. The opposite of "cheap" is "considered", not "expensive".

## Aesthetic Direction

### Theme

Light mode as the primary, opinionated default. A dark variant exists, but the design language is authored for light. This is a deliberate counter-move against the category reflex of dark mode for reader/saver tools.

### Color

- **Warm orange accent** — already the brand hue (`oklch(0.646 0.222 41.116)`). Keep it. Use it sparingly; it's a signature, not a coat of paint.
- **Tinted neutrals** — all grays carry a trace of warmth (subtle chroma toward the orange hue) so the UI feels coherent.
- **Accent is rare** — applied to the thing you should look at, not decoration.
- Avoid: cool grays, cold blue, pure black (`#000`), pure white (`#fff`).

### Typography

- **Monospace as identity.** JetBrains Mono Variable (current) or Geist Mono — both acceptable. Used throughout — UI, body, numbers — as the app's typographic voice.
- **This is a commitment.** Monospace for everything only works if it's exceptional: varied weights, tabular figures, OpenType features, deliberate tracking. If it registers as "developer tool reflex", it's failed.
- **Hierarchy is carried by weight and size**, not by swapping families.
- **Tabular figures for all numbers** — counts, dates, timestamps.
- Consider a heavier display cut (same family) for hero moments so the page doesn't read as monotone.

### Anti-references

- **Default shadcn aesthetic** — we use shadcn as primitives, but the surface must have its own voice. Bespoke spacing, typography, and details; generic card + generic button + generic input is the failure state.
- **Dashboard-ness** — hero metric layouts, sparklines as decoration, side-stripe borders on list items.
- **Generic AI tool look** — cyan-on-dark, purple gradients, glowing borders, "technical vibes" through mono alone.
- **Reader-app reflexes** — the serene dark background with serif body is a cliché in this category; we're deliberately not that.

## Design Principles

1. **Mono is a commitment, not a costume.** Using monospace for everything means it must be executed with exceptional care — tabular numbers, careful weights, deliberate kerning. If it looks like a dev tool mockup, we've failed.

2. **Light can still have depth.** Light mode without hierarchy reads as flat and templated. Create depth through weight, density, and warm neutral layers — not shadows and borders-everywhere.

3. **Density with breathing room.** Show more items than the current 4-column card grid, while giving each item enough typographic care to feel considered. Uniform grids feel templated; asymmetric rhythm feels authored.

4. **Local-first is felt, not told.** All data lives on-device via Livestore. This should translate into _instant_ interactions — live filtering, live counts, in-list fuzzy search, no spinners for things that are free. The speed is part of the brand.

5. **Keyboard-first, mouse-welcome.** Every primary action has a hotkey. The UI teaches them through subtle inline hints, not a modal cheatsheet. Power users feel at home; newcomers don't feel excluded.

6. **Details reward the attentive.** A hover reveals one additional gesture. A row expands to show what you need. Micro-interactions are small and purposeful, never showy.
